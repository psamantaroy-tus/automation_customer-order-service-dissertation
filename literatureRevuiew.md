
 1.Introduction
The migration of enterprise workloads to public cloud platforms, such as AWS, Microsoft Azure, and Google Cloud Platform, has fundamentally transformed the creation, delivery, and management of computing infrastructure. The National Institute of Standards and Technology defines cloud computing as "a model for enabling ubiquitous, convenient, on-demand network access to a shared pool of configurable computing resources … that can be rapidly provisioned and released with minimal management effort" (Mell & Grance, 2011, NIST SP 800-145). Achieving rapid provisioning at scale requires replacing manual, click-based workflows in legacy data centres with reproducible, code-based automation. The technical foundation of contemporary cloud-native engineering is Infrastructure as Code (IaC), implemented with tools such as Terraform, CI/CD pipelines, containerization, and orchestrated via Kubernetes. This literature review examines academic and industry sources relevant to the project Design, Implement and Evaluate – Automated Cloud Infrastructure Deployment Using Terraform, identifies key themes and persistent challenges, and situates the project within the current research landscape.
2. Cloud Infrastructure Provisioning: From Manual Effort to Automation
Traditional infrastructure provisioning depended on console interactions, shell scripts, and ticket-based handoffs. Sandobalin, Insfran, and Abrahão (2017) observe that "infrastructure provisioning in the cloud can be time-consuming and error-prone due to the manual process of building scripts," and note that even with configuration management tools such as Ansible, Puppet, or Chef, "[automating] the iterative development process in the cloud" remains challenging. Spinellis (2012) encapsulated this issue with the statement "Don't install software by hand" (IEEE Software, 29(4), 86–87, doi:10.1109/MS.2012.85), contending that manually configured environments lack reproducibility. These manual approaches result in four prevalent issues identified by Morris (2020) in Infrastructure as Code (O'Reilly, ISBN 978-1-098-11467-1): server sprawl, configuration drift, snowflake servers, and fragile infrastructure. Research shows that most cloud systems have a "truth" problem.
When teams make manual changes, use conflicting setup scripts, or fail to update every server, the cloud ends up looking different from what it’s supposed to. These gaps between the plan and the reality create hidden security holes.
This is why automation is critical. It’s not just about saving time; it’s about making sure your systems stay exactly as secure and consistent as you intended.
3. Infrastructure as Code: Concepts, Evolution and Tools
3.1 Definition and Evolution
Morris (2020) describes IaC as "[defining] everything as code" and using "software design and engineering practices to build your system from small, loosely coupled pieces." Rahman, Mahdavi-Hezaveh, and Williams (2019) define IaC as "the practice to automatically configure system dependencies and to provision local and remote instances," and note its use at companies like GitHub, Mozilla, Facebook, Google, and Netflix. Early IaC tools focused on configuring long-lived servers directly (Puppet 2005, Chef 2009, Ansible 2012). Later tools, such as Terraform (2014) and Pulumi (2018), moved automation to the level of entire cloud environments described in code.
3.2 Tool Landscape
Guerriero, Garriga, Tamburri, and Palomba (2019) conducted interviews with senior industry professionals and determined that Terraform is "the most popular infrastructure provisioning tool" in current practice, whereas Docker, Ansible, and Kubernetes are predominantly used for containerization and configuration management. Kumara et al. (2021) analyzed best and worst practices in IaC languages, developing a taxonomy comprising ten principal best practices and four main worst practices. Sandobalin, Insfran, and Abrahão (2020) compared the Argon tool with Ansible in experimental settings and concluded that "Argon is more effective as regards supporting the IaC approach in terms of defining the cloud infrastructure." These findings indicate that employing higher-level abstractions than Terraform's HCL may reduce errors, although such tools have not yet achieved widespread adoption.
3.3 Terraform Architecture, State and Modules
Terraform stands out for its plugin model and its steady-state file, which tracks the relationship between declared resources and actual cloud objects. Verdet, Hamdaqa, Da Silva, and Khomh (2024) report that "GitHub currently indexes 109K Terraform files with AWS configurations, compared to 35.7K with Azure and 51.5K with Google Cloud," showing Terraform's strong presence on AWS. Reusing modules is common in industry: HashiCorp's public Terraform Registry lists "6614 providers, 22072 modules & counting" (registry.terraform.io, accessed 2026). However, Sokolowski et al. (2024) found that even popular modules can have quality issues. Karanam (2024) showed that using Git for module versioning "supports automated validation, deployment, and rollback" and leads to faster disaster recovery. Still, managing state files is a challenge. Shinde (2025) lists "state file contention, secret management, and lack [of native policy enforcement]" as ongoing problems, even though Terraform "consistently achieves better performance in terms of deployment consistency, provisioning time, and code modularity" compared to CloudFormation and Ansible.
4. CI/CD Pipelines for Cloud Infrastructure
Humble and Farley's 2010 monograph Continuous Delivery: Reliable Software Releases through Build, Test, and Deployment Automation (Addison-Wesley, ISBN 978-0-321-60191-9), winner of the 2011 Jolt Excellence Award, established the canonical "deployment pipeline" as "an automated process for managing all changes, from check-in to release." For IaC, the deployment pipeline is the mechanism by which Terraform plan/apply runs are reviewed, gated and executed.
A 2025 IJIRMPS comparative study by Kumar found that GitHub Actions, when combined with Terraform and AWS CodePipeline, produced measurable "improvements in build times, deployment frequency, and mean time to recovery (MTTR)."
Many small and medium-sized cloud teams use Terraform and GitHub Actions to automate their cloud deployments. In this standard workflow, the system automatically grabs the cloud blueprint code, installs Terraform, and follows a strict routine:
Review: It cleans (fmt), verifies (validate), and previews (plan) any new changes.
Deploy: Makes the changes live (applies) once the team approves and merges the code.
Finally, a master record of the cloud's current setup is stored in a secure S3 bucket, with a DB table serving as a digital padlock to prevent developers from accidentally overwriting each other's work.
5. Containerisation and Microservices Deployment Patterns
Felter, Ferreira, Rajamony and Rubio (2014), in IBM Research Report RC25482 (AUS1407-001) An Updated Performance Comparison of Virtual Machines and Linux Containers, established empirically that "Docker containers introduce a negligible overhead for CPU and memory performance, and … applications running in a container perform equally or better when compared with traditional virtual machine technology in all tests." That result legitimised containers as the dominant unit of application packaging on cloud infrastructure provisioned by Terraform.
Di Francesco, Lago and Malavolta (2019), Architecting with microservices: A systematic mapping study (Journal of Systems and Software, vol. 150, pp. 77–97), and industrial reports establish that microservices are typically containerised, orchestrated via Kubernetes, and deployed onto IaC-provisioned virtual private clouds.
6. Environment Consistency, Configuration Drift and Immutable Infrastructure
Immutable infrastructure, which means replacing server instances rather than changing them, originated in large-scale web companies and is now described in both vendor and academic literature (Docker, Inc., 2024; Özdoğan, Ceran & Üstündağ, 2023). The main idea is that configuration drift can only be avoided if servers are never changed after deployment; any update requires creating a new image from versioned source code. Morris (2020) includes "Systems Are Disposable" and "Processes Are Repeatable" as key principles for up-to-date infrastructure. Docker images, built from versioned Dockerfiles, are the standard example of immutable artefacts, and Terraform is usually used to set up the compute resources, networking, and IAM needed around them.
Configuration drift goes against these principles. Industry studies identify four main causes: "untracked manual changes, software updates that reset configurations, inconsistent policies across teams, and the complexity of cloud and hybrid environments in which frequent provisioning creates hidden misalignments" (Reach Security, 2024). Spacelift (2024) explains that when "the desired state lives in a Git repository and the actual state lives in a cloud, you can compare them," making it possible to spot and fix drift before it causes problems.
7. Organisational Case Studies
7.1 Netflix
Basiri et al. (2016), Chaos Engineering in IEEE Software, vol. 33, no. 3, pp. 35–41 (doi:10.1109/MS.2016.60), describe Netflix's pioneering use of Chaos Monkey as well as the broader Simian Army to "[align] our teams around the notion of infrastructure robustness by isolating the problems created by server neutralisation and pushing them to the extreme." Netflix runs its entire video-delivery estate on AWS with CloudFormation- and Spinnaker-driven IaC, and its publicly documented practice of intentional fault injection presupposes that the underlying infrastructure can be re-created automatically and reproducibly.
7.2 Spotify
Spotify's published Kubernetes case study (Cloud Native Computing Foundation, 2019) records that the company was "an early adopter of microservices and Docker" with containerised microservices running across VMs since 2014 under a home-grown orchestrator (Helios), before migrating to Kubernetes. Quantitatively, "before, teams would have to wait for an hour to create a new service and get an operational host to run it in production, but with Kubernetes, they can do that on the order of seconds and minutes," and "CPU utilisation has improved on average two- to threefold." Altoros (2021) further reports that Spotify runs its "1,600+ production services" across "14 multitenant clusters in 3 regions," with "around 100 microservices being added per month."
7.3 Financial Services — Capital One
CloudBees (2018), in the case study Capital One Invests in Continuous Delivery to Automate Software Development and Deployment Activities, reports that following Capital One's adoption of a Jenkins-based continuous-delivery platform on AWS, "deployment frequency increased 1,300%," with Brock Beatty, director of software engineering at Capital One, observing that the team had moved "from a couple [of deployments] per year to now deploying every two weeks." HashiCorp documents (Storey, 2018) that Capital One uses Terraform at scale ("VPC Bones") to govern "hundreds of VPCs across multiple regions" supporting "thousands of apps" migrated out of legacy data centres.
7.4 Other Sectors
Atlassian, a widely cited mid-size example, runs its Bitbucket and Jira Cloud services on AWS infrastructure provisioned and governed by Terraform modules versioned in a monorepo. In healthcare, the heterogeneity of HIPAA-compliant deployments has so far produced few peer-reviewed Terraform case studies; the most credible cross-industry benchmarks remain those compiled by the DORA program .
8. Review Frameworks: DORA Metrics
The most influential evaluation framework for IaC-driven delivery pipelines is the set of four (subsequently five) DORA metrics — deployment frequency, lead time for changes, change failure rate and mean time to restore (MTTR), plus reliability — articulated by Forsgren, Humble and Kim in their 2018 IT Revolution monograph Accelerate: The Science of Lean Software and DevOps: Building and Scaling High Performing Technology Organizations (ISBN 978-1-942788-33-1). The empirical basis grew over subsequent years: Google Cloud's 2019 Accelerate State of DevOps Report describes itself as drawing on "six years of research and data from more than 31,000 professionals worldwide … the largest and longest-running research of its kind." DORA's annual State of DevOps report stratifies organisations into Elite, High, Medium and Low performers; the 2024 Accelerate State of DevOps Report characterises low-level teams as having "lead times of one to six months, deploying monthly to every six months," while elite teams deploy multiple times per day with lead times of less than one hour. Morris (2020) explicitly recommends the "Four Key Metrics" as the operational North Star for IaC programmes.
Forsgren et al.'s results have been independently scrutinised. Junade Ali, with the polling firm Survation, found that "other factors mattered significantly more than the outcome measures which were treated as the 'Four Key Metrics'" in surveyed software engineers and end-users (DORA Wikipedia article, citing Ali/Survation), and DORA itself in October 2023 warned against using the metrics for team-by-team performance comparisons. For an IaC project, however, DORA metrics provide an unusually clean evaluation lens, because each Terraform apply is a discrete, timestamped, auditable event from which the four indicators can be computed directly from Git and pipeline logs.
9. Research Gaps and Future Directions
Most of our information on how these tools affect real companies comes from software vendors or short interviews. Right now, there are very few independent, peer-reviewed studies that track long-term performance using standard industry metrics (DORA metrics, which measure how fast and safely a team delivers software) during a major cloud migration. Furthermore, almost no academic research focuses on highly regulated sectors like healthcare or banking, where strict security and compliance rules make these automation choices most important.
10. Synthesis and Positioning of the Project
The literature review yields five principal conclusions relevant to this project.
 First, manual provisioning is demonstrably less effective than declarative IaC across speed, error rates. 
Second, Terraform is the predominant cross-cloud IaC tool in industry, particularly on AWS, although its state file and secret management require careful administration using remote backends and KMS encryption. 
Third, CI/CD platforms—especially GitHub Actions—are sufficiently advanced to comprehensively manage Terraform workflows.
 Fourth, Docker-based containerization and microservice deployment on IaC-managed infrastructure have become standard practice, as evidenced by performance research and case studies from organisations such as Spotify and Netflix. 
Fifth, DORA metrics provide a robust and repeatable framework for evaluating these pipelines, as they can be directly measured from Terraform events.
The proposed project, Design, Implement and Evaluate – Automated Cloud Infrastructure Deployment Using Terraform, addresses three of the previously identified research gaps. By constructing a comprehensive Terraform-to-GitHub Actions-to-AWS pipeline—including EC2/ECS, IAM, S3, and security groups—and evaluating it using DORA metrics, the project generates empirical data from a complete implementation rather than relying solely on vendor case studies. Using Docker-based microservices makes it easy to test immutable infrastructure (the rule that once a piece of software is built, it is never changed—only replaced). While research shows these practices are highly recommended, not many companies are actually using them yet.
11. Conclusion
Today, writing cloud setups as code (Infrastructure as Code) is the top standard for building cloud systems, replacing older IT methods. Terraform is the most important tool in this space. However, to use it successfully, a team cannot just use it alone. They must connect it to automated shipping pipelines (CI/CD),  ensure their software containers never change once built (immutability), and constantly track their team's speed and safety using industry metrics.
The Four Pillars of Modern Cloud Setup 
IaC is now the standard, Terraform is central, but it only works well alongside -
Automated Shipping (CI/CD): Computers, not humans, test and deploy the cloud blueprints.
Built-in Security: Automated guards block security mistakes before they go live( eg : sonar cloud analysis)
Unchanging Containers: Software is locked into a permanent "box", so it behaves exactly the same way in testing as it does in production.
Metrics tracking to get performance and results.
